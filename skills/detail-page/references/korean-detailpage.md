# Korean 상세페이지 / Wadiz — conversion structure

A Korean 상세페이지 is **not a spec sheet — it's a long-scroll persuasion narrative.** Core principle: **"사람은 사진으로 끌리고 문장에 결제한다"** — images draw the eye, *copy* closes the sale. Benchmark the **order**, never copy the content. Story-structured pages convert ~2× higher than feature dumps.

## Section anatomy (consensus order)

| # | Section | Purpose | Copy pattern |
|---|---|---|---|
| 1 | **후킹 / 썸네일** | Stop the scroll in ~3s | Situation/emotion or sharp problem question, short, with a number. "보고서 하나에 3시간, 아직도 그러세요?" — NOT a product blurb. |
| 2 | **공감 (문제 제기)** | "이거 내 얘기네" | Customer's voice, concrete friction. Set the background vividly to amplify the pain. |
| 3 | **해결책** | Reframe product as the answer | Function → Result, before → after. Emphasize outcome. |
| 4 | **핵심 기능/특장점** | The proof core | 3–5 strengths, each **shown** (GIF/video for dynamic). **Benefit headline, feature as evidence underneath** (혜택→기능). |
| 5 | **사용 시나리오** | Picture owning it | Lifestyle/model shots + "이럴 때 쓰세요" contexts (출근길에, 주말 캠핑에서). |
| 6 | **스펙/상세** | Answer rational questions | Tables for numeric/comparative. Sensitive concerns (safety): image → table → Q&A. |
| 7 | **신뢰 요소** | Remove hesitation | Cumulative counts, real reviews, Before/After, 임상/특허/KC인증, 수상, 미디어. Data + third-party, not self-praise. **Repeat mid-scroll, not only at bottom.** |
| 8 | **비교표** | Justify "왜 우리" | Us vs alt / vs the old way, checkmark table, few honest rows. |
| 9 | **혜택/가격** (Wadiz: **리워드 구성**) | Present the deal | Anchor value before price; show savings. Wadiz: reward table 구성·가격·수량. |
| 10 | **일정/배송** | Set expectations | Dates, batch vs sequential, courier + CS number, maker contact, 도서산간 note. |
| 11 | **메이커 스토리/브랜드** | Earn belief | Why it was made; maker photo + real name. **Wadiz: move to the FRONT (right after the hook).** |
| 12 | **FAQ** | Knock down last objections | Real buyer questions; packaging/storage, 도서산간, 1–2 platform/reward FAQs. |
| 13 | **환불·교환·AS** | Compliance + risk reversal | Conditions, windows, who pays shipping. Clear & fair raises conversion. |
| 14 | **최종 CTA** | Convert now | Recap ONE benefit + urgency directly above the button. Place CTAs throughout, not just the bottom. |

Collapsed frames:
- **5-block (commerce):** 후킹 → 문제공감 → 해결제안 → 신뢰확보 → 행동유도(CTA).
- **4-stage (Wadiz):** 오프닝(상상 질문 + 메이커 이유) → 브릿지(핵심 강점 3–5개, 가장 강한 것 먼저, 여기서 계속 읽을지 결정됨) → 본문(증명) → 클로징(전환).

## Pre-flight lens — Schwartz Awareness × Sophistication (decide BEFORE PASONA)
PASONA hard-codes a *problem-aware* opening, which is wrong for a retargeted or already-convinced reader. Before writing, set two brief fields (Eugene Schwartz, *Breakthrough Advertising*):

**Awareness — where the reader is, which sets where the hero STARTS:**
- **Unaware** → open with story/symptom/identity; problem not yet named. (PASONA from the very top.)
- **Problem-aware** → open by naming the pain vividly. (Classic PASONA.)
- **Solution-aware** → open with the **mechanism / "왜 이게 다른가"**; skip heavy agitation.
- **Product-aware** → open with proof, differentiation, offer; they know you, need a reason to pick you.
- **Most-aware** (retargeted, returning) → open with the **offer/price/urgency**; minimal warm-up.

**Sophistication — how tired the market is, which sets the CLAIM style (the direct antidote to AI-slop sameness):**
- **Stage 1–2** (new/young market) → a plain benefit/bigger-claim headline still works.
- **Stage 3** → lead with a **unique mechanism** (the "how" no one else is saying).
- **Stage 4–5** (claims exhausted) → mechanism is also tired → pivot to a **new identity / experience / sensory framing**; "best/strongest" is dead, so do NOT write another superlative.

Use these to choose the hero's starting block and headline type; then apply PASONA from that entry point. Two slivers as brief inputs (`awareness`, `sophistication`), zero model dependency.

## Copywriting backbone — PASONA (use as the skeleton)
**P**roblem(문제) → **A**ffinity(공감) → **S**olution(해결 *원리*, 제품 언급 X) → **O**ffer(구체적 제품/제안) → **N**arrow(대상 좁히기 "특히 ~한 분께") → **A**ction(긴급 CTA).
The #1 mistake: collapsing **Solution** (the mechanism) into **Offer** (the packaged product) — selling too early triggers reader defensiveness. Also referenced: AIDA, FAB (Feature→Advantage→Benefit). **Entry point is set by the Awareness lens above** — a Most-aware reader skips straight toward Offer/Action.

> **Scope note — compliance is OUT of this skill.** Plan for quality only. 표시·광고법 / 추천·보증(뒷광고) substantiation varies by platform and project, so the **user owns it**. Don't add claim-source gates or disclosure checks to the planner, prompts, or rubric. (Sophistication 4–5 already steers away from bare superlatives for *quality* reasons, not legal ones.)

## 혜택-before-기능 rule
(1) list features → (2) derive the customer's advantage → (3) re-express as a **benefit in the customer's words**. The feature becomes evidence under a benefit headline.

## Headline templates (reusable)
- Problem question: `[명사구], 아직도 그러세요?` → "보고서 하나에 3시간, 아직도 그러세요?"
- Number + benefit: `[숫자] + [혜택]` → "한 번의 클릭으로 5개 상세페이지를 완성하세요."
- Curiosity + ease: `[궁금증]? 이미 준비돼 있습니다. [간편 행동]만 하세요.`
- Benefit + urgency + command: `[혜택], 지금 시작하세요.`
- Before→After: `[기능] → [결과]` → "단어 몇 개 입력 → 보고서 초안 자동 생성."
- Offer + deadline (CTA): `지금 [행동] 시 [혜택1] + [혜택2] / [마감일]까지.`

## Vocabulary
- **Urgency/scarcity:** 마감임박, 한정수량, 선착순, 단 N개, 오늘까지, 이번 한정. Two-digit caps read as more "real" than huge caps.
- **Trust:** 정품, KC인증/인증, 안전결제, 무료배송, 누적 판매/리뷰, Before&After, 임상시험, 특허, 수상, 미디어 보도. Specific non-round numbers read as more credible.
- **Tone:** conversational, emotional, sensory — "수선 없이 바로 입어도 살아나는 핏" beats "핏 좋은 데님입니다." The whole page should leave **one** memorable message.

## Wadiz specifics (crowdfunding ≠ open market — explain *why the maker made this*)
Mandatory story blocks that gate platform review:
1. **메이커 소개** — real name + bright photo, at the very front.
2. **제작 동기 / 개발 스토리** — why this had to exist.
3. Product story: 오프닝 → 브릿지(강점 3–5) → 본문(증명).
4. **리워드 구성** — table of 구성·가격·수량, mid-story.
5. **배송 정보** — method, batch vs sequential, courier + CS, maker contact, 도서산간.
6. **FAQ** + **환불·교환·AS 정책** (per-project).

**Reward ladder (the core conversion lever — first 72h ≈ 54% of funding):**
- 울트라 얼리버드 > 슈퍼 얼리버드 > 얼리버드 > 일반, each progressively less discounted.
- Main reward ideally 70,000원대+ to lift per-backer value; same composition cheaper, or same price + extra.
- Quantity caps are **immutable after launch**; two-digit caps raise participation; don't oversize early-bird caps or other tiers won't fund.
- Naming: `[얼리버드 뱃지] + [브랜드/제품명] + [구성]`, consistent; consumables state usage period ("3개월 팩").
- Refund baseline (state per-project): cancellable until maker registers shipping or within 7 days of delivery; defect/mis-ship = maker pays shipping, change-of-mind = supporter pays.

## Wadiz AI / digital-product pattern

For AI education, automation systems, templates, prompt packs, PDF/VOD/Figma files, and creator/business education, load `wadiz-ai-digital-benchmark.md` and enforce these additions:

- **Make the intangible tangible:** show covers, Figma frames, prompt sheets, spreadsheet libraries, VOD modules, icon packs, open-chat access, challenge calendars. A digital product that is only described feels overpriced.
- **Proof every claim fast:** after a strong claim, place an evidence object within the next viewport: previous funding screenshot, review, tool screen, before/after, package checklist, or curriculum extract.
- **Use a bad-example block:** show what the buyer currently does wrong, mark it with arrows/X/handwritten notes, then explain the mechanism that fixes it.
- **Separate mechanism from offer:** first sell the reusable mechanism (`AI봇`, `공식`, `3분 생성`, `디자인 X AI`), then sell the package.
- **Repeat package economics in the story body:** Wadiz reward cards are not enough. Body sections must show 구성, 원가/정가, 할인율, 추천 패키지, 수량/기간.
- **Long curriculum as value proof:** chapter lists, day-by-day missions, bonus modules, update policy. This justifies price after the emotional sell.
- **Authority stack near the top and mid-scroll:** category rank, accumulated funding, previous projects, maker credentials, media/awards, follower/community count.

Minimum module stack for this category:
1. Hero proof thumbnail
2. Authority / previous funding proof
3. Pain or bad-example contrast
4. Mechanism reveal
5. Live tool/demo screenshot
6. Review or result wall
7. Deliverables wall
8. Package economics
9. Event/urgency CTA

## Layout & size conventions
- Long vertical scroll, **image-dominant**, **mobile-first**. Text often baked into full-width image bands — but never burn critical headline text into the hero where mobile makes it unreadable; avoid thin fonts / faint colors / tight letter-spacing (they smear on mobile).
- Content width: **Naver/Wadiz 860px**, Coupang 780px, 11번가 831px — **860px is the safe default**. Build images at **1000–1200px** wide and downscale for retina crispness.
- Source-canvas title ~60–80px; body large enough to survive mobile downscaling. Test everything at mobile size.
- Background-removed cutouts → PNG; photos → JPEG. Demonstrate dynamic features with **GIF / short video**.
- Separate each persuasion block as a distinct full-width band with whitespace / thin rule / recurring section-number motif so the long scroll stays navigable.

## Korean typography (Fontshare is Latin-only — this is a real gap)
A Korean page needs a Korean webfont; the Latin families in `aesthetics.md`/Fontshare don't render 한글. Pair a Korean face with a Latin accent face for numerals/labels.
- **Body (KR sans, the professional default — not slop):** **Pretendard** (`cdn.jsdelivr.net/gh/orioncactus/pretendard`), or IBM Plex Sans KR, Gowun Dodum, Spoqa Han Sans Neo.
- **Display (KR):** Noto Serif KR (editorial/premium), Nanum Myeongjo, Gowun Batang (serif); Gmarket Sans, Black Han Sans, Jua (bold sans display — use with restraint).
- **Latin accent** (eyebrows, numerals, prices — Korean detail pages lean on tabular Latin numerals): Fraunces, Clash Display, a mono, etc. Use `font-variant-numeric: tabular-nums` for prices/stats.
- Load order: Pretendard via its CDN `<link>` + the KR display + Latin accent via Google Fonts. Set `--body`/`--kr-serif`/`--lat` CSS vars.
- Mobile legibility: avoid thin KR weights and tight letter-spacing — 한글 smears at small sizes more than Latin. Body ≥ 16px.

## Mobile-first responsiveness (the shoot.js mobile shot catches this)
Korean detail pages are bought on phones. A multi-column desktop layout MUST collapse to one column at ~640–680px. Pitfall: inline `style="grid-template-columns:..."` does NOT respond to media queries unless overridden with `!important` (inline beats stylesheet) — prefer Tailwind responsive utilities (`grid-cols-1 md:grid-cols-2`) or a media-query block with `!important`. Empty image/photo panels collapse to 0 height when stacked — give them a `min-height`. Always verify with `scripts/shoot.js` and read `mobile-full.png` before scoring.

## Pre-publish checklist (detail mode)
- [ ] First screen hooks in ≤3s with one sharp message, not a product blurb.
- [ ] Empathy block in the customer's voice; pain is concrete.
- [ ] Every feature preceded by a benefit headline (혜택→기능).
- [ ] Whole page reduces to **one** memorable message.
- [ ] Real numbers/social proof, credible; trust repeated mid-scroll.
- [ ] Dynamic features shown via GIF/video, not described.
- [ ] Urgency directly above each CTA; multiple CTAs down the page.
- [ ] Built 1000–1200px → exported to platform width; mobile type legible; no critical text burned into hero.
- [ ] (Wadiz) maker intro at front + why-made story; reward table w/ tiers & quantities; delivery, FAQ, refund all present (review-gating).
- [ ] (Wadiz) reward ladder + limited two-digit quantities; consumables show usage period; naming convention consistent.
- [ ] (AI/digital) At least 8 benchmark modules from `wadiz-ai-digital-benchmark.md`; digital deliverables are visible artifacts, not abstract bullets.
- [ ] (AI/digital) There is a before/after or bad-example contrast block plus a mechanism block before pricing.
